01 Fase 1 — Reconhecimento com nmap
01
Descobrir portas e serviços do alvo
O nmap é o scanner de rede mais usado do mundo. Vamos ver quais "portas" estão abertas no alvo e o que está rodando nelas.
⏱ 15 min
🎯 O QUE VOCÊ VAI DESCOBRIRQuais portas estão abertas, qual serviço roda em cada uma e qual versão do software está sendo usada.
1.1
Instalar as ferramentas no Kali
KALI — root@kali#
root@kali#apt update -qq
← aguarde terminar, depois rode o próximo
KALI — root@kali#
root@kali#apt install -y nmap curl apache2-utils
← aguarde terminar (~2 min). Vai aparecer muito texto, é normal.
apt update atualiza a lista de programas disponíveis.
apt install -y nmap curl apache2-utils instala as ferramentas da aula. O apache2-utils inclui o ab (Apache Benchmark) que usaremos na Fase 3.
1.2
Varredura básica — quais portas estão abertas?
KALI — root@kali#
root@kali#nmap localhost
Starting Nmap scan...
PORT STATE SERVICE
8080/tcp open http-proxy ← nosso alvo!
nmap localhost verifica as 1000 portas mais comuns da sua máquina. open significa que a porta está aberta e aceitando conexões.
1.3
Descobrir a versão do serviço — flag -sV
KALI — root@kali#
root@kali#nmap -sV -p 8080 localhost
PORT STATE SERVICE VERSION
8080/tcp open http Apache httpd 2.4.x ← versão revelada!
-sV pede para o nmap descobrir a versão exata do programa rodando.
-p 8080 foca só na porta 8080, assim fica mais rápido.
💡 POR QUE A VERSÃO IMPORTA?Cada versão de software tem vulnerabilidades públicas catalogadas. Saber a versão é o primeiro passo para saber como atacar — ou como defender.
1.4
Scan completo — salvar resultado
KALI — root@kali#
root@kali#nmap -sV -sC -p 8080 localhost -oN /tmp/scan-nmap.txt
| http-title: Damn Vulnerable Web Application
| http-methods: GET POST HEAD OPTIONS
| http-auth: Basic realm="DVWA"
-sC roda scripts automáticos: descobre título, métodos HTTP e mais.
-oN /tmp/scan-nmap.txt salva o resultado — você vai precisar disso no relatório!

02 Fase 2 — Inspeção manual com curl
02
Ver o que o servidor revela nos cabeçalhos HTTP
O curl faz requisições HTTP e mostra exatamente o que o servidor responde. É como inspecionar o "envelope" das mensagens entre navegador e servidor.
⏱ 15 min
🎯 O QUE VOCÊ VAI DESCOBRIROs cabeçalhos HTTP revelam tecnologias usadas, versões de software e proteções de segurança ausentes — tudo sem precisar abrir o navegador.
2.1
Ver os cabeçalhos do servidor
KALI — root@kali#
root@kali#curl -I http://localhost:8080
HTTP/1.1 200 OK
Server: Apache/2.4.x ← versão exposta!
X-Powered-By: PHP/7.x ← linguagem exposta!
← SEM: Strict-Transport-Security
← SEM: X-Frame-Options
← SEM: Content-Security-Policy
-I busca apenas os cabeçalhos, sem baixar toda a página.
Cada linha SEM: indica um cabeçalho de segurança ausente — isso é uma vulnerabilidade real!
⚠️ CABEÇALHOS AUSENTES = VULNERABILIDADETodo cabeçalho de segurança que falta é um achado. Anote no relatório com criticidade Médio.
2.2
Ver o código HTML da página de login
KALI — root@kali#
root@kali#curl http://localhost:8080/login.php
← aparece o HTML da página de login
Baixa o HTML da página de login. Procure as tags form, input e action — elas mostram como o formulário está estruturado.
2.3
Testar login com credenciais padrão

Este comando tem várias partes. Digite tudo em uma linha só no terminal:

KALI — root@kali# (tudo em uma linha)
root@kali#curl -X POST http://localhost:8080/login.php -d "username=admin&password=password&Login=Login" -v 2>&1 | grep "Location"
Location: index.php ← login aceito! Credencial padrão funcionou!
-X POST envia dados como formulário.
-d "username=admin&password=password" são as credenciais testadas.
Se aparecer Location: index.php, o sistema aceitou a senha padrão!
✅ ACHADO CRÍTICO — CREDENCIAL PADRÃO ATIVAO sistema aceita admin / password sem exigir troca. Criticidade CRÍTICO no relatório.
2.4
Salvar evidências do curl
KALI — root@kali#
root@kali#curl -I http://localhost:8080 > /tmp/scan-curl.txt
← salvo em /tmp/scan-curl.txt
O > redireciona a saída para um arquivo. Este arquivo será evidência no relatório.
03 Fase 3 — Simulação de DoS (negação de serviço)
03
Sobrecarregar o servidor com muitas requisições
Um ataque DoS (Denial of Service) tenta derrubar um serviço enviando mais requisições do que ele consegue processar. Aqui vamos simular isso dentro do nosso container.
⏱ 15 min
🎯 O QUE VOCÊ VAI APRENDER Como um servidor reage quando recebe requisições demais ao mesmo tempo — e por que aplicações sem proteção ficam lentas ou param de responder.
⚠️ APENAS NO CONTAINER LOCAL O comando abaixo funciona exclusivamente contra localhost:8080 — o container que você mesmo subiu. Nunca aponte para qualquer outro endereço. DoS em servidores externos é crime.
3.1
Verificar o servidor respondendo normalmente

Antes do ataque, abra o navegador em http://localhost:8080 e veja o site carregando normalmente. Deixe essa aba aberta para comparar depois.

KALI — root@kali#
root@kali#curl -o /dev/null -s -w "%{time_total}\n" http://localhost:8080
0.043 ← tempo de resposta normal em segundos (menos de 1s)
Este comando mede o tempo de resposta do servidor. Guarde esse número — você vai comparar com o tempo durante o ataque.
-o /dev/null descarta o conteúdo, -w "%{time_total}" mostra só o tempo total.
3.2
Disparar o ataque DoS com Apache Benchmark (ab)

O ab (Apache Benchmark) é uma ferramenta legítima de teste de carga. Vamos usá-la para enviar muitas requisições simultâneas ao servidor. Digite tudo em uma linha:

KALI — root@kali# (tudo em uma linha)
root@kali#ab -n 5000 -c 100 http://localhost:8080/
This is ApacheBench, Version 2.3
Benchmarking localhost (be patient)...
Completed 500 requests
Completed 1000 requests...
Concurrency Level: 100
Time taken for tests: X.XXX seconds
Requests per second: XXX.XX [#/sec]
Failed requests: XX ← servidor começou a falhar!
-n 5000 = número total de requisições a enviar (5 mil).
-c 100 = número de requisições simultâneas (100 ao mesmo tempo).

Enquanto o ab roda, tente recarregar o site no navegador — ele vai estar lento ou sem responder. Isso é o efeito do DoS na prática!

Preste atenção na linha Failed requests: ela mostra quantas requisições o servidor não conseguiu atender.
🔥 O QUE ESTÁ ACONTECENDO? O Apache tem um limite de conexões simultâneas (MaxRequestWorkers). Quando todas as "vagas" estão ocupadas, o servidor começa a recusar novas conexões — é exatamente isso que um ataque DoS explora. Em vez de hackear o código, o atacante simplesmente esgota os recursos do servidor.
3.3
Medir o impacto — tempo de resposta durante o ataque

Abra um segundo terminal do Kali (em outro PowerShell, rode o Comando 2 novamente) e execute enquanto o ataque ainda está rodando no primeiro terminal:

KALI — segundo terminal — root@kali#
root@kali#curl -o /dev/null -s -w "%{time_total}\n" http://localhost:8080
5.231 ← tempo de resposta DURANTE o ataque (muito maior!)
Compare os dois tempos medidos:
Antes do ataque: < 0.1 segundo
Durante o ataque: vários segundos ou timeout

Essa diferença é o impacto de disponibilidade — métrica principal num relatório de DoS.
3.4
Aumentar a intensidade (opcional)

Se o servidor aguentou bem, tente com mais requisições simultâneas:

KALI — root@kali# (tudo em uma linha)
root@kali#ab -n 10000 -c 200 http://localhost:8080/
← servidor deve começar a rejeitar mais requisições
-n 10000 = 10 mil requisições totais.
-c 200 = 200 conexões simultâneas.

Observe o resultado: quantas Failed requests? O servidor chegou a parar de responder completamente?
3.5
Salvar resultados para o relatório
KALI — root@kali# (tudo em uma linha)
root@kali#ab -n 5000 -c 100 http://localhost:8080/ > /tmp/scan-dos.txt
← resultado completo salvo em /tmp/scan-dos.txt
Salva o resultado completo do ab em arquivo — inclua no relatório como evidência da Fase 3.

🛡️ COMO SE DEFENDER — CONTRAMEDIDAS PARA ATAQUES DoS

DEFESAO QUE FAZONDE CONFIGURAR
Rate LimitingLimita quantas requisições por IP por segundomod_evasive no Apache, nginx limit_req
Timeout baixoEncerra conexões lentas antes de travar o servidorTimeout e KeepAliveTimeout no Apache
WAFFirewall que detecta e bloqueia padrões de ataqueModSecurity, Cloudflare WAF
CDN / Load BalancerDistribui o tráfego entre vários servidoresCloudflare, AWS CloudFront, nginx upstream
MonitoramentoAlerta quando o tráfego sobe anormalmenteGrafana, Prometheus, Datadog
